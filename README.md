# RaveSpin website

Static showcase site for [RaveSpin](https://github.com/marc-rene/RaveSpin). Build with **Node** or **Bun**, then copy the built files to your GitHub Pages repo (e.g. your personal site or a project page).

## Build (Node or Bun)

From this folder (`Documentation/Website`):

```bash
# With Node (v18+)
npm install
npm run build

# With Bun
bun install
bun run build
```

Output is in **`dist/`**. The contents of `dist/` are what you deploy (copy to your GitHub Pages repo root).

## Local preview

```bash
npm run preview   # or: bun run preview
```

Serves the built site from `dist/` so you can check links and layout before deploying.

## Deploying to GitHub Pages

### Option A: Copy to your Personal GitHub Pages repo

1. Build: `npm run build` or `bun run build`
2. Copy **everything inside** `Documentation/Website/dist/` into the root of your `username.github.io` repo.
3. Commit and push. Your site will be at `https://username.github.io`.

If your site is in a **project repo** (e.g. `username.github.io/RaveSpin`), you need to serve from a subpath:

1. In `vite.config.js`, set `base: '/RaveSpin/'` (replace `RaveSpin` with your repo name if different).
2. Rebuild and copy the contents of `dist/` to the root of the branch/folder that GitHub Pages uses (e.g. `gh-pages` branch or `/docs`).

### Option B: GitHub Actions (automatic deploy from this repo)

A workflow is provided to build this site and deploy to the **`gh-pages`** branch of the same repo. After the first run, enable GitHub Pages in the repo settings:

1. **Settings â†’ Pages**
2. **Source:** Deploy from a branch
3. **Branch:** `gh-pages` / root

Then every push to `main` (or the branch you use) will rebuild and update the site. The workflow file is in the repo root: `.github/workflows/deploy-website.yml`.

If you use **project pages** (e.g. `marc-rene.github.io/RaveSpin`), the workflow sets `base: '/RaveSpin/'` so assets load correctly. For a **user/org site** (e.g. `username.github.io`), you would copy the built files to that repo manually (Option A) or add a second workflow that pushes to your user site repo.
